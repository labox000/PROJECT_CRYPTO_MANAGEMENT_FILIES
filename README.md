## Overview

This application allows multiple users to store files **confidentially** on a shared
machine. Each user has a password-protected account and can encrypt their files using
an algorithm of their choice before saving them. An **administrator** (the first
registered user) supervises the system and has access to activity logs and user
management tools.

The key challenge of this project was implementing **every cryptographic primitive
from scratch** in Python — SHA-1, SHA-256, DES, 3DES, AES-128, ChaCha20, RSA and
ElGamal — without using any external cryptographic library.

## Features

###  User
- Sign up / Sign in with strong password policy enforcement
- Automatic first-user → admin promotion
- Change username (requires current password confirmation)
- Change password (requires old + new + confirmation)
- Add, open, modify and delete personal files
- Choose encryption algorithm at upload time
- Verify file integrity at any time
- View other users' file names (read-only, content inaccessible)
- Automatic session expiry (30 minutes)

###  Administrator
- Dedicated admin panel (auto-detected at login)
- List all users with UUID, username, role and creation date
- View sensitive details: password hash + salt for each user
- Modify any user's username
- Reset any user's password (without knowing the old one)
- Delete any user (cascades to their files and sessions)
- View complete activity logs (all users)
- Filter logs by username or user ID
- View all files across all users

## Cryptographic Algorithms

All algorithms are implemented **manually** in `core/crypto.py`.

### Hashing
| Algorithm | Output | Usage |
|-----------|--------|-------|
| SHA-1     | 160 bits | File mode (pedagogical, one-way) |
| SHA-256   | 256 bits | Password hashing, file integrity |

### Symmetric Encryption
| Algorithm | Mode | Key size | Notes |
|-----------|------|----------|-------|
| DES       | CFB  | 8 bytes  | PKCS#7 padding, 16 Feistel rounds |
| 3DES      | CFB  | 24 bytes | EDE mode (Encrypt-Decrypt-Encrypt) |
| AES-128   | CBC  | 16 bytes | Full S-Box, MixColumns in GF(2⁸), KeyExpansion |
| ChaCha20  | Stream | 32 bytes | 12-byte nonce, 20 rounds (RFC 7539) |

### Asymmetric Encryption
| Algorithm | Key size | Notes |
|-----------|----------|-------|
| RSA       | 512 bits | Miller-Rabin primality test, e=65537, block encryption |
| ElGamal   | 256 bits | Miller-Rabin, g=2, (c₁, c₂) pair per block |

> **Note:** RSA and ElGamal are provided for educational purposes and are suited only
> for small data volumes (key exchange, small messages).

### Password Security
Passwords are stored as `SHA-256(salt ‖ password)` where the salt is a **32-byte
random value** generated by `secrets.token_bytes()`. Salt and hash are stored
separately in the database. This makes rainbow table attacks computationally
infeasible.

### File Integrity
Before encryption, a **SHA-256 hash of the plaintext content** is computed and stored
in the database (`Files.integrity_hash`). On every file open, the decrypted content is
re-hashed and compared. Any tampering — even a single modified byte — triggers an
integrity warning.

## Project Structure

```
project/
│
├── main.py               # Entry point — auto-routes to user or admin menu
├── schema.sql            # SQLite schema: Users, Sessions, Files, Logs
├── README.md
├── User_Management.db    # SQLite database (auto-generated at runtime)
│
├── core/
│   ├── __init__.py
│   ├── authentification.py   # Register, login, sessions, account management
│   ├── crypto.py             # All cryptographic algorithms (from scratch)
│   ├── db.py                 # SQLite data access layer (CRUD)
│   ├── files.py              # Encrypted file operations
│   └── objects.py            # User, File, Session data classes
│
└── storage/
    └── users/
        ├── <user_id>/        # Encrypted files stored per user UUID
        └── ...
```

### Database Schema

```
Users    (user_id, username, password_hash, salt, role, created_at, updated_at)
Sessions (session_id, user_id, created_at, expires_at)
Files    (file_id, owner_id, filename, stored_path, crypto_mode,
          crypto_meta, integrity_hash, created_at, updated_at)
Logs     (log_id, user_id, action, detail, created_at)
```

All foreign keys use `ON DELETE CASCADE`. Triggers automatically update
`updated_at`. Logs have **no delete function** exposed — they are append-only.

## Getting Started

### Prerequisites

```bash
Python 3.10+
```

No external dependencies required — only Python standard library.

### Installation

```bash
git clone https://github.com/<your-username>/<repo-name>.git
cd <repo-name>
```

### First Run

```bash
python main.py
```

On first launch, `init_db()` creates the SQLite database from `schema.sql`.
**The first user to register automatically becomes the administrator.**

> ⚠️ To reset the application entirely (including all users and files), delete
> `User_Management.db` and the `storage/` directory, then re-run `main.py`.

## Usage

### Application Flow

```
python main.py
    │
    ├── 1. Sign in  ──► role detected ──► User menu  OR  Admin panel
    └── 2. Sign up  ──► first user = admin, others = user
```

### Password Policy

Passwords must contain:
- At least **6 characters**
- At least one **uppercase** letter
- At least one **lowercase** letter
- At least one **digit**
- At least one **special character** (`!@#$%^&*()-_=+...`)

### File Encryption Flow

```
User selects file path
        │
        ▼
Read raw bytes
        │
        ▼
Compute SHA-256 integrity hash  ◄── stored in DB
        │
        ▼
User selects algorithm (SHA1 / SHA256 / DES / 3DES / AES / ChaCha20 / RSA / ElGamal)
        │
        ▼
encrypt(data, mode)  ──► generates random key + IV/nonce
        │
        ▼
Save encrypted bytes to storage/users/<user_id>/<filename>
        │
        ▼
Store metadata (crypto_mode, crypto_meta JSON, integrity_hash) in DB
```

### Logged Actions

Every significant event is recorded with timestamp, username and detail:

| Action | Trigger |
|--------|---------|
| `register` | New account created |
| `login` | Successful login |
| `login_failed` | Failed login attempt |
| `logout` | User disconnected |
| `add_file` | File added and encrypted |
| `modify_file` | File content updated |
| `delete_file` | File deleted |
| `verify_integrity` | Integrity check (result included) |
| `change_username` | Username modified |
| `change_password` | Password updated |
| `delete_user` | Admin deleted a user |

## Security Design

| Threat | Mitigation |
|--------|-----------|
| Password leak from DB | SHA-256 + 32-byte random salt per user |
| Unauthorized file access | `owner_id` check on every file operation |
| File tampering | SHA-256 integrity hash verified on every open |
| SQL injection | Parameterized queries throughout `db.py` |
| Weak passwords | Enforced policy (uppercase, digit, special char) |
| Session hijacking | One active session per user, 30-minute expiry |
| Log tampering | No delete/update function exposed for `Logs` table |
| First-user privilege | First registered account automatically gets `admin` role |

## Performance

Measured on a standard machine (Intel i5, 8 GB RAM, SSD):

### Symmetric algorithms — 10 MB file

| Algorithm | Encrypt | Decrypt |
|-----------|---------|---------|
| ChaCha20  | 0.22 s  | 0.22 s  |
| AES-128   | 0.28 s  | 0.27 s  |
| DES       | 0.41 s  | 0.42 s  |
| 3DES      | 0.95 s  | 0.96 s  |

### Asymmetric algorithms — small data (pedagogical)

| Algorithm | Key generation | Encrypt | Decrypt | Volume |
|-----------|---------------|---------|---------|--------|
| RSA 512-bit | — | 2.80 s | 12.30 s | 1 KB |
| ElGamal 256-bit | ~18 ms | ~1.3 ms | ~1.0 ms | 100 bytes |
| ElGamal 256-bit | ~18 ms | ~5.4 ms | ~5.0 ms | 500 bytes |

> RSA decryption is slow because the private exponent `d` is much larger than
> `e = 65537`. ElGamal is fast on small data but **doubles the ciphertext size**
> (two integers c₁ and c₂ per block).
